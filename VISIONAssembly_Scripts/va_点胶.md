# 凌云光 VISIONAssembly 工程流程梳理文档（CCD1/CCD2 通用）
## 一、工程整体架构总览
你提供的多张流程图截图，本质是**单路 CCD 视觉检测工程**的完整流程拆解（CCD2 与 CCD1 逻辑完全一致，仅工具编号/参数差异）。整体流程按**「图像采集→预处理→定位/标定→检测→结果输出/存储」**工业视觉标准链路设计，共分为 6 大核心阶段，每个阶段分工明确，逻辑闭环。

### 整体流程层级（从左到右/从上到下）
```
【图像采集阶段】→ 【预处理阶段】→ 【核心定位/标定阶段】→ 【检测分析阶段】→ 【结果输出/存储阶段】→ 【循环/收尾阶段】
```
以下按阶段拆解，结合截图中工具节点逐一解析。

## 二、各阶段详细梳理（含工具含义+逻辑说明）
### 阶段1：图像采集与基础配置（最左侧节点）
**核心目标**：触发相机采集图像，完成相机参数、触发模式、曝光等基础配置，保证图像质量稳定。
#### 关键工具/节点（截图左侧可见）：
| 工具名称（截图可见） | 核心作用 | 小白理解 |
|----------------------|----------|----------|
| 相机触发工具 | 发送触发信号，控制相机拍照（软触发/硬触发） | 让相机“拍一张图”，是所有流程的起点 |
| 相机参数工具 | 配置曝光、增益、ROI、触发延迟等相机参数 | 调相机“画质”，比如曝光时间调长/缩短 |
| 图像格式工具 | 配置图像格式（灰度/彩色、分辨率、像素格式） | 规定图像“存储格式”，比如灰度图（8位） |
| 图像保存/路径配置 | 配置原始图像保存路径、命名规则（按时间/批次） | 拍好的图存哪、叫什么名 |

**逻辑说明**：此阶段是**流程入口**，所有后续操作依赖图像质量。CCD1/CCD2 仅相机IP/通道不同，此阶段逻辑完全一致。

### 阶段2：图像预处理（承接采集阶段的核心净化）
**核心目标**：对原始图像去噪、校正、增强，消除干扰（反光、灰尘、噪点），为后续检测提供“干净图像”。
#### 关键工具/节点（截图左侧第二列）：
| 工具名称 | 核心作用 | 小白理解 |
|----------|----------|----------|
| 灰度变换工具 | 图像灰度值拉伸/压缩、二值化、反转 | 调图像“对比度”，让目标更清晰 |
| 滤波工具 | 高斯滤波、中值滤波、均值滤波 | 去图像“噪点/毛刺”，比如相机噪点、灰尘斑点 |
| 畸变矫正工具 | 矫正相机镜头畸变（桶形/枕形） | 修正图像“扭曲”，比如镜头边缘变形 |
| ROI裁剪工具 | 截取图像感兴趣区域（跳过无关区域） | 只留“要检测的部分”，比如只拍零件区域，跳过背景 |
| 图像融合/差分 | 多帧图像融合、背景差分（针对动态场景） | 消除动态干扰，比如流水线移动的背景杂波 |

**逻辑说明**：预处理是“洗图”，**必须保证目标特征（如边缘、斑点、Mark点）清晰可辨**，否则后续检测必出误判。此阶段是视觉工程的“地基”，CCD1/CCD2 预处理逻辑一致，仅参数（如滤波核大小、ROI坐标）微调。

### 阶段3：核心定位/标定阶段（流程核心枢纽）
**核心目标**：完成**单目标定+手眼标定**（你之前学的核心知识），确定目标位置、坐标系转换关系，为后续检测/加工提供精准坐标。
#### 关键工具/节点（截图中间偏左，分支汇合节点多）：
| 工具名称 | 核心作用 | 对应你之前的知识 |
|----------|----------|------------------|
| 几何定位工具 | 提取目标边缘、轮廓、特征点，计算中心/角度/尺寸 | 对应“单目标定+定位”，提取Mark点/目标中心 |
| 相机标定工具 | 单目平面标定（像素→世界坐标） | 对应你之前的“单目标定”，建立像素与毫米的关系 |
| 手眼标定工具 | 相机坐标→机器人/激光/运动平台坐标 | 对应你之前的“手眼标定”，让视觉坐标能被设备执行 |
| 搜索区域工具 | 配置定位搜索ROI（遍历Cell/多区域） | 对应你之前的“遍历Cell”，限定定位范围 |
| 结果个数/总像素数工具 | 统计定位结果数、Blob像素数 | 你之前写脚本用的“number/Pixels”参数来源 |
| 分支/汇合节点 | 控制流程分支（比如定位成功/失败分支） | 工业脚本的“if/else”逻辑载体 |

**核心逻辑（结合你之前写的脚本）**：
此阶段通过**几何定位工具**提取目标特征，用**脚本工具**（你之前写的OK/NG判断、NG下标存储）统计结果，最终通过**手眼标定**将“视觉坐标”转换为“设备可执行坐标”。
**难点理解**：分支汇合节点多（如“分支→几何定位→分支汇合→循环”），本质是**多区域遍历（如30个Cell）**，每个分支对应一个Cell的定位/判断，最终通过“分支汇合”汇总所有结果。

### 阶段4：检测分析阶段（流程核心，占比最大）
**核心目标**：对预处理后的图像，结合定位结果，进行**缺陷检测、尺寸测量、特征匹配、逻辑判断**，最终判定每个Cell/目标的OK/NG状态。
#### 关键工具/节点（截图中间列，表达式计算多、分支多）：
| 工具名称 | 核心作用 | 小白理解 |
|----------|----------|----------|
| Blob工具 | 提取连通域（Blob），统计像素数、面积、周长、个数 | 你之前脚本用的“Blob工具_005”，判断目标个数/像素是否达标 |
| 形状匹配工具 | 模板匹配（标准模板→待测目标，判断偏差） | 对比目标是否“形状一致”，比如零件是否变形 |
| 边缘提取工具 | 提取目标边缘，计算边缘距离、角度、直线度 | 测量尺寸，比如孔距、边距、直线度 |
| 灰值分析工具 | 分析图像灰度值（均匀性、对比度、缺陷斑点） | 检测表面缺陷，比如污点、划痕、亮斑 |
| 表达式计算 | 自定义逻辑判断（如像素数>1000且个数=4→OK） | 你之前写的脚本逻辑，核心是“OK/NG判定” |
| 几何测量工具 | 测量圆度、同心度、垂直度、距离 | 精密尺寸检测，比如零件圆度是否合格 |
| 特征提取工具 | 提取纹理、颜色、角点等特征 | 复杂缺陷检测，比如纹理不一致、颜色偏差 |

**核心逻辑**：
此阶段是**工程的“大脑”**，结合阶段3的定位结果，通过“Blob+几何+灰值+脚本判断”多维度检测，最终给每个目标打“OK/NG”标签。
**你之前的脚本适配点**：你写的“NG下标存储、最后统一显示”脚本，就是嵌入在此阶段的**循环分支中**（每个Cell检测完成后，将NG下标存入全局列表，最后一次循环统一显示）。

### 阶段5：结果输出/存储阶段（流程右侧）
**核心目标**：将检测结果（OK/NG、坐标、缺陷类型、图像）输出给设备（如PLC、机器人、激光），并存储图像/数据用于追溯。
#### 关键工具/节点（截图最右侧列）：
| 工具名称 | 核心作用 | 工业意义 |
|----------|----------|----------|
| 结果输出工具 | 输出OK/NG信号、坐标、偏差值给PLC/运动平台 | 告诉设备“合格与否、要走的坐标” |
| 文本生成工具 | 生成检测结果文本（如“Cell1-OK”“Cell2-NG”） | 图像上显示检测结果，你之前脚本的“编号显示”依赖此工具 |
| 图像保存/路径配置 | 保存OK/NG图像（按分类存储，如NG/OK/日期） | 工业追溯必备，方便现场排查缺陷 |
| CSV数据导出 | 导出检测数据（时间、Cell号、OK/NG、尺寸值）到Excel | 统计分析，比如批次合格率、缺陷率 |
| 路径规划/运动控制 | 结合定位坐标，规划设备运动路径（激光/点胶/搬运） | 激光切割、点胶、搬运的核心执行逻辑 |
| 循环控制/分支 | 控制多批次/多目标循环检测，或分支处理NG/OK | 流程“循环”的核心，比如遍历30个Cell |

### 阶段6：循环/收尾阶段（最下方）
**核心目标**：循环执行检测流程（多目标/多批次），并完成收尾操作（如清空缓存、复位状态）。
#### 关键工具/节点：
| 工具名称 | 核心作用 |
|----------|----------|
| 循环工具 | 控制循环次数/触发条件（如遍历所有Cell、循环检测） |
| 并行汇合/分支汇合 | 等待所有分支任务完成，进入下一阶段 |
| 复位工具 | 清空全局变量（如NG下标列表、结果列表）、缓存图像 |
| 异常处理 | 捕获异常（如相机断连、工具报错），触发报警/重试 |

## 三、CCD1/CCD2 工程差异与通用逻辑
### 1. 完全一致的部分
- 流程结构：6大阶段完全一致，工具类型完全一致；
- 核心逻辑：采集→预处理→定位→检测→输出→循环，工业视觉标准流程；
- 脚本逻辑：你写的“NG下标存储、最后统一显示”脚本，可直接复用（仅需修改工具编号/变量名）。

### 2. 仅有的差异点
- 硬件配置：CCD1/CCD2 对应不同相机（IP/通道）、不同光源/镜头；
- 局部参数：预处理参数（滤波、ROI）、定位参数（搜索区域、模板）、检测阈值（像素数、尺寸公差）因产品/工位差异微调；
- 工具编号：截图中工具后缀（如_001、_002）因工程顺序不同略有差异，但工具功能一一对应。

## 四、核心难点与不好理解的点（小白/新人常见困惑）
结合你之前的开发需求，梳理出**工程中最易困惑的6个点**，并给出通俗解释：

### 难点1：多分支/汇合节点混乱（阶段3/4最明显）
**困惑点**：截图中大量“分支→工具→分支汇合”，分不清执行顺序。
**通俗解释**：
- 分支=“if判断”（比如定位成功走分支1，失败走分支2）；
- 分支汇合=“等待所有分支完成”（比如30个Cell的定位都做完，再进入检测阶段）；
- 类比：你遍历30个Cell，每个Cell是一个分支，所有Cell都处理完，才会“汇合”进入下一步。

### 难点2：全局变量/列表的生命周期（你之前的NG列表）
**困惑点**：为什么要用`GvVisionAssembly.ng_index_list`，而不是局部列表？
**通俗解释**：
- 脚本在**循环内运行**，每次循环都执行，局部列表每次都会清空，存不住所有NG下标；
- 全局变量（挂在`GvVisionAssembly`下）生命周期贯穿整个工程，能保存所有循环的NG结果；
- 最后一次循环时，用全局列表对比，统一显示结果，符合“循环存结果、最后一次展示”的工业逻辑。

### 难点3：坐标转换的串联（单目标定→手眼标定→设备执行）
**困惑点**：为什么要做两次标定（单目+手眼），直接用图像坐标不行吗？
**通俗解释**：
- 单目标定：把“图像像素坐标”转成“相机世界坐标（毫米）”；
- 手眼标定：把“相机世界坐标”转成“机器人/激光/运动平台坐标”；
- 类比：你用手机（相机）拍地图（图像），单目标定是把地图像素转成实际公里数，手眼标定是把公里数转成汽车（设备）的行驶路线。

### 难点4：最后统一显示的逻辑（循环+延迟）
**困惑点**：为什么不在每次循环都显示，而要等最后一次？
**通俗解释**：
- 每次循环显示会闪烁、卡顿，影响现场观察；
- 最后一次显示，能一次性看到所有Cell的OK/NG结果，直观判断整体缺陷分布；
- 工业现场追求“结果清晰、无干扰”，这是量产工程的标准设计。

### 难点5：Blob工具与表达式计算的配合
**困惑点**：Blob工具的“总像素数”“结果个数”怎么和脚本结合？
**通俗解释**：
- Blob工具：提取目标的“像素总数”“目标个数”（比如零件有4个孔，结果个数=4）；
- 表达式计算/脚本：设置阈值（如像素数≤1000且个数=4→OK），判断是否合格；
- 核心是“用工具统计特征，用逻辑判断合格标准”。

### 难点6：图像路径/存储的多级分类
**困惑点**：为什么要分“原图/OK图/NG图/日期路径”，直接存不行吗？
**通俗解释**：
- 工业追溯：出问题时，能快速找到对应时间、对应工位的图像；
- 数据管理：避免图像堆积，按日期/批次分类，方便清理；
- 这是视觉工程的“标配操作”，所有量产项目都这么设计。

## 五、工程优化建议（基于工业现场实战）
结合你之前的开发需求，给你3个可落地的优化点，提升工程稳定性/可读性：
### 优化1：脚本模块化（拆分复杂脚本）
你现在的脚本逻辑（NG下标存储、显示）可拆分为2个脚本：
- 脚本1（循环内）：仅负责“判断NG、存下标”；
- 脚本2（最后一次循环）：仅负责“读取NG下标、画框、显示”；
好处：逻辑清晰，后期修改显示逻辑不用动判断逻辑。

### 优化2：增加异常兜底
在脚本开头/关键节点增加异常捕获（如`try-except`），避免单一步骤报错导致整个流程中断：
```python
try:
    # 核心逻辑（获取数据、存NG列表、画图）
except Exception as e:
    GvVisionAssembly.ReportMessage(f"报错：{str(e)}", GvVisionAssembly.GeMsgReportType.eMRTError, False)
    # 触发报警/重试
```

### 优化3：参数可视化配置
将关键参数（如NG阈值、Cell总数、显示偏移量）通过`GvVar`设置为全局变量，在流程中可视化配置，不用改代码：
```python
# 全局变量配置（在流程中设置#nThreshold、#totalCount）
threshold = int(GvVar.GetVar("#nThreshold"))  # NG阈值
total_count = int(GvVar.GetVar("#totalCount")) # 总Cell数
```
